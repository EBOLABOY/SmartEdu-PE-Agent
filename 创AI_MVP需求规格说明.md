# 创AI MVP需求规格说明：动屏智创

## 1. MVP目标

本文件定义第一版可开发、可演示、可参赛的最小可行产品。

项目名称：**《动屏智创：小学体育课堂互动大屏协同创作智能体的开发与应用》**

项目类别：**创AI案例 / 教育智能体**

MVP不是万能系统，而是完成一条闭环：

```text
教师输入课堂需求
-> AI生成教案和互动大屏初稿
-> 教师通过多轮对话局部修改
-> 系统实时预览互动大屏
-> 教师手动微调关键内容
-> 导出单文件HTML大屏和结构化项目数据
-> 用真实体育课堂场景完成演示和证据采集
```

MVP必须证明三件事：

1. 系统能解决小学体育课堂组织中的真实问题。
2. 系统不是一次性生成器，而是支持多轮协同修改的智能体。
3. 系统生成的大屏不是静态课件，而是可运行、可交互、可投屏的课堂工具。

## 2. 参赛主案例

MVP主案例固定为：**三年级篮球运球接力挑战课**。

基础设定：

```text
学段：小学
年级：三年级
学科：体育
班级人数：40人
课时长度：40分钟
场地：半个篮球场
器材：篮球20个、标志桶8个
课堂类型：技能练习 + 游戏竞赛
核心技能：行进间运球、绕桶、接力配合
育人目标：规则意识、团队合作、公平竞争、安全运动
```

选择这个案例的原因：有规则讲解、路线演示、计时计分、分组轮换、安全提醒和团队合作表达空间，容易真实落地并采集证据。

## 3. MVP边界

MVP不做：

- 不做全学科通用教案生成器。
- 不做所有体育项目的万能平台。
- 不做视频动作识别。
- 不做学生体质健康预测。
- 不做自动体育成绩评定。
- 不采集学生姓名、人脸、照片、体测敏感数据。
- 不让AI直接自由生成完整HTML。
- 不做复杂动画编辑器。
- 不做完整手机遥控端。
- 不做教研组云端共享。

MVP只做：

- 小学体育课堂组织场景。
- 教案与互动大屏协同生成。
- 多轮对话局部修改。
- 稳定模板渲染。
- 单文件HTML导出。
- 一个主案例的完整演示闭环。

## 4. 用户流程

### 4.1 创建课堂项目

教师输入基础信息：

```text
年级：三年级
人数：40人
教学内容：篮球运球接力挑战
场地：半个篮球场
器材：篮球20个、标志桶8个
课时：40分钟
目标：提升行进间运球能力，培养团队合作和规则意识
```

系统生成教案初稿、课堂流程、互动大屏配置、分组方案、游戏规则、路线图、倒计时、计分器和安全提醒。

### 4.2 预览初稿

教师查看左侧AI对话、中间项目结构、右侧大屏预览，并判断规则是否清楚、时间是否合理、分组是否可执行、路线是否安全、大屏是否适合投屏。

### 4.3 多轮对话修改

教师继续输入修改需求：

```text
规则太复杂了，改成二年级也能听懂的版本。
增加四组计分，每组10人，队名用动物名称。
把热身改成3分钟球性练习。
路线图里增加绕桶返回路线，强调从右侧返回。
加一个安全提醒：返回队伍时从外侧绕行，避免碰撞。
把整体颜色改成篮球主题，按钮更大一点，适合投屏。
```

系统行为：AI理解修改意图，输出结构化修改补丁，系统校验补丁，更新Project State，保存版本，重新渲染预览。

### 4.4 手动微调

教师可以不用AI，直接手动修改队名、比分、倒计时时长、规则文字、安全提醒和课堂环节标题。

### 4.5 导出与课堂使用

教师导出单文件HTML互动大屏、Project State JSON、教案Markdown草稿和使用说明。课堂中投屏打开HTML，切换课堂环节，启动倒计时，修改比分，展示路线图，显示安全提醒。

## 5. 页面布局

MVP采用三栏布局：

```text
左侧：AI对话修改区
中间：项目结构与编辑区
右侧：互动大屏预览区
```

AI对话修改区必须包含对话历史、教师输入框、发送按钮、AI建议摘要、本次修改影响范围提示、应用修改按钮、放弃修改按钮。

项目结构与编辑区必须包含项目信息、教学目标、课堂流程、大屏模块列表、队伍设置、安全提醒、版本记录、导出按钮。

互动大屏预览区必须包含当前大屏预览、全屏按钮、投屏模式按钮、模块切换、倒计时操作、计分操作。

## 6. P0功能清单

### 6.1 初稿生成

输入字段：年级、人数、教学内容、场地、器材、课时长度、教学目标。

输出内容：教案结构、大屏配置、分组方案、规则说明、路线图配置、计时计分模块、安全提醒。

验收标准：

- 能基于主案例生成完整Project State。
- 生成内容符合小学体育课堂基本逻辑。
- 大屏预览能正常渲染。

### 6.2 多轮局部修改

首版必须支持这些修改操作：

```text
update_text      修改文字
update_time      修改时间
update_teams     修改队伍
update_rules     修改规则
update_route     修改路线图
update_theme     修改主题
add_module       添加模块
remove_module    删除模块
reorder_section  调整课堂流程
regenerate_part  重写某个局部
```

验收标准：

- 修改倒计时时长时，不影响规则和队伍。
- 修改队伍时，不影响路线图。
- 修改主题时，不影响内容结构。
- AI改坏后可以回退。

### 6.3 实时预览

每次Project State变化后，大屏预览自动刷新。

验收标准：规则、倒计时、队伍、路线、安全提醒等模块修改后，预览同步变化。

### 6.4 版本回退

每次应用AI修改前保存版本。

版本记录至少包含版本ID、修改时间、修改说明、Project State快照。

验收标准：可以回退到上一个版本，回退后预览同步恢复。

### 6.5 手动编辑

首版支持手动编辑队名、比分、倒计时时长、规则文字、安全提醒文字、课堂环节名称。

验收标准：手动修改立即更新Project State，并立即刷新预览。

### 6.6 单文件HTML导出

导出的HTML必须能离线运行。

要求：

- 不依赖外网CDN。
- CSS和JS内嵌。
- 支持全屏。
- 支持基础交互。
- 支持16:9投屏。

验收标准：双击HTML可打开，倒计时可运行，计分器可修改，路线图可显示，安全提醒可显示。

## 7. 大屏模块定义

### 7.1 规则说明模块 rules

用途：让学生快速理解游戏规则。

```json
{
  "id": "rules-main",
  "type": "rules",
  "title": "游戏规则",
  "items": ["每组依次运球绕桶返回", "掉球后原地捡球继续", "返回后拍手交接"]
}
```

### 7.2 课堂流程模块 flow

用途：展示课堂环节。

```json
{
  "id": "flow-main",
  "type": "flow",
  "title": "课堂流程",
  "steps": [
    {"name": "球性热身", "duration": 3},
    {"name": "运球练习", "duration": 10},
    {"name": "接力挑战", "duration": 15},
    {"name": "放松总结", "duration": 5}
  ]
}
```

### 7.3 倒计时模块 timer

用途：控制热身、练习、比赛时间。

```json
{
  "id": "timer-game",
  "type": "timer",
  "title": "接力挑战倒计时",
  "duration": 300
}
```

### 7.4 计分器模块 scoreboard

用途：记录队伍比分。

```json
{
  "id": "score-main",
  "type": "scoreboard",
  "title": "小组积分",
  "teams": [
    {"name": "猎豹队", "score": 0},
    {"name": "雄鹰队", "score": 0},
    {"name": "海豚队", "score": 0},
    {"name": "骏马队", "score": 0}
  ]
}
```

### 7.5 分组轮换模块 rotation

用途：展示小组轮换或练习站变化。

```json
{
  "id": "rotation-main",
  "type": "rotation",
  "title": "轮换安排",
  "groups": ["第一组", "第二组", "第三组", "第四组"],
  "instruction": "听到哨声后按顺时针方向轮换"
}
```

### 7.6 路线图模块 route_map

用途：展示运球路线、接力路线或战术路线。

```json
{
  "id": "route-main",
  "type": "route_map",
  "title": "运球接力路线",
  "field": "half_basketball_court",
  "objects": [
    {"type": "cone", "x": 40, "y": 35},
    {"type": "cone", "x": 60, "y": 35}
  ],
  "routes": [
    {"from": [20, 50], "to": [60, 35], "label": "绕桶前进"},
    {"from": [60, 35], "to": [20, 50], "label": "右侧返回"}
  ]
}
```

### 7.7 安全提醒模块 safety

用途：强调体育课堂安全约束。

```json
{
  "id": "safety-main",
  "type": "safety",
  "title": "安全提醒",
  "items": ["绕桶时保持间距", "返回时从队伍外侧绕行", "掉球后先观察再捡球"]
}
```

### 7.8 体育精神模块 spirit

用途：展示规则意识、合作、公平竞争等提示。

```json
{
  "id": "spirit-main",
  "type": "spirit",
  "title": "今日体育精神",
  "items": ["遵守规则", "为队友加油", "胜不骄，败不馁"]
}
```

    "grade": "三年级",

## 9. 多轮修改协议

AI每次修改不直接返回完整项目，而是返回修改补丁。

### 9.1 修改响应格式

```json
{
  "summary": "将比赛时间改为8分钟，并新增安全提醒模块。",
  "riskLevel": "low",
  "requiresConfirmation": false,
  "operations": [
    {
      "op": "replace",
      "path": "/screenConfig/modules/timer-game/duration",
      "value": 480
    },
    {
      "op": "add",
      "path": "/screenConfig/modules/-",
      "value": {
        "id": "safety-main",
        "type": "safety",
        "title": "安全提醒",
        "items": ["绕桶时保持间距", "返回时从队伍外侧绕行"]
      }
    }
  ]
}
```

### 9.2 风险等级

```text
low       文字、主题、队名、时间等低风险修改
medium    课堂流程、路线、分组方式等会影响课堂组织的修改
high      安全规则、对抗强度、器材使用、场地路线等高风险修改
```

高风险修改必须要求教师确认。

### 9.3 操作校验

应用补丁前必须校验：

- `path` 是否存在。
- `module.type` 是否合法。
- `duration` 是否为正数。
- `teams` 是否非空。
- `safety.items` 是否非空。
- `route_map` 坐标是否在画布范围内。

## 10. 导出要求

MVP至少支持四类导出。

### 10.1 单文件HTML

用于课堂投屏。CSS和JS必须内嵌，不依赖外网，支持全屏、倒计时、计分和路线图展示。

### 10.2 Project State JSON

用于复现和继续编辑。必须包含 `metadata`、`lessonPlan`、`screenConfig`、`versions`，并使用UTF-8编码。

### 10.3 教案草稿

用于开发与应用报告或课堂使用。首版可导出Markdown，后续再考虑Word。

### 10.4 应用证据包

用于参赛材料，建议包含初始需求、AI生成结果、多轮修改记录、版本记录、最终Project State、最终HTML。

## 11. 演示视频流程

### 11.1 问题引入

展示体育课堂痛点：规则讲解难、计时计分繁琐、路线说明抽象、动态课件制作耗时。

### 11.2 初稿生成

输入主案例：

```text
生成一节三年级篮球运球接力挑战课，40人，20个篮球，半个篮球场，重点训练行进间运球和团队合作。
```

展示系统生成教案结构、大屏模块、规则、路线图、倒计时和计分器。

### 11.3 多轮修改

连续展示3到5轮修改：

```text
把规则改得更适合三年级学生。
增加四组计分，每组10人。
路线图增加右侧返回，避免碰撞。
加入安全提醒。
改成篮球主题投屏风格。
```

重点展示：不是重写全部，而是局部更新；预览实时变化；可以回退版本。

### 11.4 课堂使用

展示导出的HTML大屏：全屏投屏、倒计时启动、比分增加、路线图展示、安全提醒展示。

### 11.5 应用效果

展示课堂照片、教师反思、学生反馈、修改记录和导出资源。

## 12. 第一版开发任务

建议按这个顺序开发：

```text
1. 创建项目骨架
2. 定义Project State类型
3. 实现主案例默认数据
4. 实现三栏页面布局
5. 实现大屏模块渲染
6. 实现手动编辑
7. 实现版本记录与回退
8. 实现模拟AI修改补丁
9. 实现真实AI接入
10. 实现单文件HTML导出
11. 实现教案Markdown导出
12. 准备演示数据和课堂应用证据
```

开发顺序原则：先跑通Project State到大屏渲染，再做手动编辑，再做版本回退，再接AI。不要一开始就依赖大模型，否则调试会乱。

## 13. 验收标准

MVP完成的最低标准：

1. 可以创建“三年级篮球运球接力挑战课”。
2. 可以生成教案结构和大屏配置。
3. 可以渲染规则、流程、倒计时、计分、分组、路线图、安全提醒。
4. 可以通过至少5条自然语言指令修改已有内容。
5. 修改是局部更新，不破坏其他模块。
6. 可以回退版本。
7. 可以手动修改队名、比分、时间、规则。
8. 可以导出单文件HTML并离线打开。
9. 可以导出Project State JSON。
10. 可以录制一条完整演示视频。

## 14. 参赛材料对应关系

MVP功能与参赛要求的对应关系：

```text
教育智能体：Web Agent协同创作工作台
AI赋能开发：AI生成结构化配置和修改补丁
有效解决问题：解决体育课堂组织和动态资源制作难题
操作简单易用：自然语言修改 + 大屏模板渲染 + 单文件导出
真实落地：三年级篮球运球接力课堂应用
开源分享：提供源代码、Project State示例、模板和使用说明
资料完整：报告、视频、HTML、JSON、截图、课堂证据
```

## 15. 最终结论

MVP只需要把一件事做扎实：

> 让小学体育教师通过多轮对话，生成并不断修改一节真实体育课可用的互动大屏。

第一版成功的标志不是功能多，而是：

```text
结构稳定
修改可控
投屏可用
证据完整
主案例讲得清楚
```

如果MVP能做到这一点，就足够支撑参赛开发、演示视频和开发与应用报告。
