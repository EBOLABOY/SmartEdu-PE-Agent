# 创AI参赛项目总纲：跃课

## 1. 项目最终定位

项目名称暂定：

**《跃课：小学体育课堂互动大屏协同创作智能体的开发与应用》**

项目类别：

**创AI案例 / 教育智能体**

项目一句话定义：

> 本项目面向小学体育课堂组织效率低、动态资源制作难、课堂互动反馈弱等真实问题，开发支持多轮对话修改的体育课堂互动大屏协同创作智能体，实现教案、大屏、计时计分、路线演示、安全提醒和课后反思的一体化生成与迭代应用。

更短的表达：

> 一个能和小学体育教师反复对话、共同打磨课堂教案和互动大屏的 AI 工作台。

项目本质不是“AI教案生成器”，不是“体育PPT生成器”，也不是“万能HTML网页生成器”。

项目本质是：

```text
小学体育课堂组织问题
-> AI理解教师需求
-> 生成结构化课堂方案
-> 渲染成互动大屏
-> 多轮对话局部修改
-> 真实课堂使用
-> 运行记录与课后反思
-> 沉淀为可复用模板
```

## 2. 核心问题

小学体育课堂高度依赖现场组织。教师在有限课时内要同时完成规则讲解、队伍调度、技能练习、计时计分、安全提醒和课堂评价。

传统口头讲解和静态PPT存在三个真实痛点：

1. **讲不清**：体育游戏、战术路线、练习规则靠口头讲解，低年级学生理解慢。
2. **管不过来**：分组轮换、倒计时、计分、安全提醒都靠教师人工控制，课堂负担重。
3. **做不起**：教师想用互动大屏，但每节课单独制作HTML或动态课件，成本太高。

本项目解决的不是“教师不会做网页”，而是：

> 体育教师缺少一种低成本、可修改、可投屏、可复用的体育课堂互动资源生成与协同创作工具。

## 3. 核心判断

这个项目值得做，因为它同时满足：

- 有真实教学场景：小学体育课堂组织。
- 有学科特色：规则讲解、路线演示、计时计分、分组轮换、安全提醒。
- 有AI开发价值：多轮协同创作、结构化生成、局部修改、模板渲染。
- 有技术壁垒：不是一次性生成文本，而是维护Project State并生成可运行互动大屏。
- 有应用证据空间：可以真实上一节课，采集课堂使用照片、截图、运行记录和教师反思。
- 有发展路线：可扩展到模板库、手机控制端、教研共享、校本体育资源库。

必须避免的错误方向：

- 不做全学科通用教案生成器。
- 不做万能HTML网页生成器。
- 不做复杂视频动作识别。
- 不做学生体质健康预测。
- 不做自动体育成绩评定。
- 不让AI直接自由生成不可控HTML。
- 不把课程思政做成口号墙。
- 不把所有体育项目一开始都支持完。

## 4. 目标用户与使用场景

目标用户：

- 小学体育教师。
- 体育教研组。
- 需要组织体育游戏、体能练习、球类活动、接力竞赛和课堂展示的教师。

主应用场景建议：

**三年级篮球运球接力挑战课**

原因：

- 小学体育常见。
- 有规则讲解需求。
- 有路线演示需求。
- 有计分和倒计时需求。
- 有团队合作和体育精神表达空间。
- 容易真实落地并采集应用证据。

配套演示场景：

1. **体能循环练习**：展示倒计时、轮换、任务站、安全提醒。
2. **篮球运球接力**：展示规则说明、路线图、计分器、团队合作。
3. **简易战术/路线演示**：展示移动路线、协同配合和课堂共创。

## 5. 产品形态

系统是一个Web Agent协同创作工作台，建议采用三栏布局：

```text
左侧：AI对话修改区
中间：课堂项目结构区
右侧：互动大屏实时预览区
```

### 5.1 左侧：AI对话修改区

教师通过自然语言与系统对话，例如：

```text
生成一节三年级篮球运球接力课，40人，20个篮球，半个篮球场。
规则太复杂了，改成二年级也能听懂的版本。
增加四组计分，每组10人，队名用动物名称。
加一个3分钟热身倒计时，内容是球性练习。
路线图里增加绕桶返回路线，强调从右侧返回。
把整体颜色改成篮球主题，按钮大一点，适合投屏。
```

### 5.2 中间：课堂项目结构区

展示并允许手动编辑：

- 课题名称。
- 年级与人数。
- 场地与器材。
- 教学目标。
- 课堂流程。
- 大屏模块。
- 队伍设置。
- 安全提醒。
- 版本记录。

### 5.3 右侧：互动大屏实时预览区

实时预览课堂投屏效果，包括：

- 课堂流程。
- 游戏规则。
- 倒计时。
- 计分器。
- 分组轮换。
- 动作要点。
- 安全提醒。
- 合作、公平竞争、坚持挑战等体育精神提示。
- 简易路线图或战术演示。

## 6. 核心设计原则

### 6.1 AI不直接自由生成HTML

错误设计：

```text
教师输入需求
-> 大模型直接生成完整HTML
-> 祈祷按钮能用
```

正确设计：

```text
教师输入需求
-> AI生成结构化JSON配置
-> 系统校验JSON
-> 模板引擎渲染HTML
-> 教师预览和修改
-> 导出HTML大屏
```

理由：

- 自由生成HTML不稳定。
- 容易破坏之前已修改内容。
- 按钮、计时器、计分器可能失效。
- 样式不可控，不利于投屏。
- 不便于复现和维护。

### 6.2 系统维护唯一真相源Project State

所有教案、大屏、模块、版本都来自统一的Project State。

示例结构：

```json
{
  "metadata": {},
  "lessonPlan": {},
  "screenConfig": {},
  "conversationHistory": [],
  "versions": [],
  "runtime": {},
  "reflection": {},
  "template": {},
  "safetyRules": []
}
```

### 6.3 多轮对话只产生局部修改补丁

AI不应每次重写整个项目，而是输出局部修改操作。

示例：

```json
{
  "operations": [
    {
      "op": "replace",
      "path": "/screenConfig/modules/timer/duration",
      "value": 480
    },
    {
      "op": "replace",
      "path": "/screenConfig/modules/scoreboard/teams",
      "value": ["红队", "蓝队", "黄队", "绿队"]
    },
    {
      "op": "add",
      "path": "/screenConfig/modules/-",
      "value": {
        "id": "safety",
        "type": "safety",
        "title": "安全提醒",
        "items": ["绕桶时保持间距", "返回后从队伍外侧回到队尾"]
      }
    }
  ]
}
```

### 6.4 教师拥有最终控制权

AI只能建议，不能替教师决定。

尤其是以下内容必须可由教师确认或手动修改：

- 安全规则。
- 场地路线。
- 队伍组织。
- 对抗强度。
- 器材使用。
- 课堂评价。

## 7. 核心功能清单

### 7.1 P0：参赛MVP必须完成

1. **初稿生成**
   - 教师输入年级、项目、人数、场地、器材、教学目标。
   - 系统生成教案和互动大屏配置。

2. **多轮对话修改**
   - 支持教师用自然语言修改已有内容。
   - 例如修改规则、队伍、时间、路线、安全提醒、主题风格。

3. **局部更新**
   - AI输出修改补丁。
   - 系统只更新相关模块，不重写全部内容。

4. **实时预览**
   - 每次修改后，大屏预览立即变化。

5. **版本回退**
   - 每次AI修改形成一个版本。
   - 修改失败或效果不好时可以回退。

6. **手动编辑**
   - 教师可以直接修改队名、时间、规则、比分、标题。
   - 不能让所有操作都依赖AI。

7. **导出单文件HTML**
   - 大屏可以离线运行。
   - 不依赖外网CDN。
   - 适合学校投屏环境。

8. **基础大屏模块**
   - 规则说明。
   - 课堂流程。
   - 倒计时。
   - 计分器。
   - 分组轮换。
   - 路线图。
   - 安全提醒。

9. **投屏模式**
   - 大字体。
   - 高对比。
   - 16:9自适应。
   - 一键全屏。
   - 大按钮。
   - 键盘快捷键控制倒计时和比分。

10. **安全提醒结构化**
    - 体育课安全必须内置为结构化约束，而不是普通文本。

### 7.2 P1：强烈建议完成

1. **临场调整模式**
   - 支持下雨、器材不足、人数变化、时间压缩、场地变化等体育课堂真实情况。

2. **课堂运行记录**
   - 记录倒计时使用、比分变化、环节完成情况、教师临场修改。

3. **课后反思生成**
   - 根据课堂运行记录和教师反馈，生成课后反思。

4. **模板库**
   - 把优秀课例沉淀为可复用模板。

5. **体育项目知识库**
   - 篮球、跳绳、立定跳远、体能循环、接力游戏等。

6. **应用证据导出**
   - 导出生成记录、修改记录、课堂运行记录和反思，用于参赛报告。

### 7.3 P2：发展路线

1. **手机遥控端**
   - 体育教师不可能一直站在电脑前，手机控制是未来刚需。

2. **学生共创入口**
   - 学生可参与队名、规则、挑战任务的共创。

3. **教研组共享**
   - 同校体育组共享模板和课例。

4. **多模态资源输入**
   - 上传现有教案、图片、场地图，生成大屏。

5. **动作诊断融合**
   - 未来可与动作技能诊断项目结合。

6. **校本体育资源库**
   - 沉淀学校自己的游戏、规则、评价表、安全规范。

## 9. 技术架构建议

推荐架构：

```text
教师输入层
-> 需求表单 + 自然语言对话

AI Agent层
-> 初稿生成Agent
-> 修改理解Agent
-> 安全校验Agent
-> 反思生成Agent

结构化数据层
-> Project State
-> JSON Patch
-> Version History
-> Template Library

渲染层
-> Lesson Renderer
-> Screen Renderer
-> Exporter

课堂应用层
-> 投屏大屏
-> 计时计分
-> 路线演示
-> 运行记录
```

关键原则：

- Prompt不是架构，数据结构才是架构。
- AI生成配置，程序负责渲染。
- 先保证稳定，再追求花哨。
- 页面必须适合真实投屏，不是只适合截图。

## 10. 安全与伦理设计

体育课安全是硬约束。

必须考虑：

- 不采集学生姓名、人脸、照片、体测敏感数据。
- 队伍使用红队、蓝队、黄队、绿队等匿名名称。
- 涉及跑跳投、对抗、器械、队伍路线的内容必须有安全提醒。
- AI生成的活动建议必须由教师确认。
- 高风险修改需要二次确认。
- 大屏内容不应制造学生排名羞辱或身体能力歧视。
- AI生成的文本、图片、视频、音频、动画等，应按比赛要求标记“AI生成”。

## 11. 课堂应用闭环

冲高奖不能只停留在生成和导出。

必须形成完整闭环：

```text
生成
-> 修改
-> 预览
-> 课堂使用
-> 运行记录
-> 课后反思
-> 沉淀模板
-> 下次复用
```

这才是从工具升级为系统。

课堂应用证据至少包括：

- 教师初始需求截图。
- AI生成初稿截图。
- 多轮修改记录截图。
- 大屏预览截图。
- 投屏使用照片。
- 课堂活动照片。
- 倒计时、计分、路线图运行截图。
- 教师课后反思。
- 学生反馈。
- 修改前后版本对比。
- 导出的HTML文件。
- 配置JSON。
- 使用说明。

## 12. 评委可能质疑与回答

### 12.1 这和普通AI生成PPT有什么区别？

回答：

> 本系统不是生成静态课件，而是生成可交互、可修改、可运行的课堂大屏。它支持计时、计分、轮换、路线演示和多轮对话局部修改，能直接服务体育课堂现场组织。

### 12.2 AI生成HTML不稳定怎么办？

回答：

> 系统不让AI直接自由生成HTML，而是让AI生成结构化配置和修改补丁，再由固定模板渲染，保证核心交互稳定。

### 12.3 体育课上老师哪有空操作？

回答：

> 系统采用投屏大按钮、快捷键和预设流程。教师课前生成，课中只需切换环节、启动倒计时和调整比分。后续可扩展手机控制端。

### 12.4 有没有真实应用？

回答必须靠证据：

> 使用一节真实小学体育课作为应用案例，提供生成记录、修改记录、课堂投屏照片、运行记录、教师反思和学生反馈。

### 12.5 是否只适合个人使用？

回答：

> 系统支持模板复用和导出单文件HTML，其他教师只需输入年级、人数、器材和项目即可生成自己的课堂大屏。

## 14. 最终结论

当前项目方向确定为：

> 小学体育课堂互动大屏协同创作智能体。

项目必须体现三句话：

```text
不是一次生成，而是多轮协同创作。
不是静态课件，而是课堂可运行大屏。
不是单节课工具，而是可沉淀、可复用、可发展的体育教学智能体。
```

最高等级奖项的关键不是标题大，而是问题真、结构清、系统稳、证据硬。

本项目的第一性原则：

```text
用最稳定的数据结构，让AI帮助体育教师生成、修改、运行和沉淀真实课堂资源。
```
